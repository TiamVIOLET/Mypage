## 1. 背景と全体的な位置づけ

複雑な業務プロジェクトでは、AIプログラミングを一回限りのプロンプトだけに頼るのは安定しません。より堅実な方法は、プロジェクトルール、ドメイン知識、Skillシステム、タスクワークフロー、Subagentの分担、自動コンテキスト注入、品質ゲート、知識の蓄積を組み合わせて、ひとつの協働体系にすることです。

この体系の目的は明確です。AIがコードを書く前にプロジェクトを理解すること。AIが思いつきではなく手順に沿って進めること。複雑なタスクを分解、再開、追跡できること。そして調査、振り返り、合意事項を継続的にプロジェクト資産として残すことです。

全体としては、次のような層に分けられます。

```text
AIプログラミング体系
├── プロジェクト入口層：CLAUDE.md / AGENTS.md
├── 知識ドキュメント層：.claude/docs / モジュールCLAUDE.md
├── Skill層：.claude/skills
├── コマンド層：.claude/commands
├── Agent分担層：.claude/agents
├── Trellisワークフロー層：workflow / tasks / spec / workspace
├── 自動注入層：hooks / settings / エディタhooks
└── 品質とアーカイブ層：review / test / update-spec / commit / finish-work
```

## 2. プロジェクト入口層

プロジェクト入口層は、AIに「このプロジェクトは何か」「境界はどこか」「どの全体ルールに従うべきか」を伝えます。

ルートの`CLAUDE.md`は、通常三つの役割を持ちます。第一に、技術スタック、実行方法、主要モジュール、重要な業務オブジェクトなどのプロジェクト構成を説明します。第二に、どのディレクトリがアプリケーションホストで、どこが共有層で、どの依存関係が許可され、どれが禁止されるかといったレイヤー境界を示します。第三に、AIの作業ルール、たとえば要件を推測しない、既存パターンを優先する、変更前に文書を読む、完了後に関連知識を更新する、といった制約を定義します。

モジュール単位の`CLAUDE.md`は、局所的なルールを説明するのに向いています。たとえば、特定モジュールの命名規則、ディレクトリ構造、よく使うサービス、テスト入口、過去の約束事などです。AIがそのサブディレクトリに入ったとき、全体ルールよりも精密なコンテキストを得られます。

`AGENTS.md`は、ツール横断の補助入口として使えます。コーディングエージェントによって既定で読むファイルが異なるため、重要なルールを汎用入口にも同期しておくと、モデルやツールを切り替えたときのコンテキスト損失を減らせます。

## 3. 知識ドキュメント層

`.claude/docs/`は、安定したプロジェクト知識を保存する場所です。アーキテクチャ説明、モジュールマップ、よくある問題、調査記録、振り返り、過去の意思決定などを置きます。その価値は「人が読む」だけではありません。次のタスクでAIが素早くコンテキストを復元できることが重要です。

ドキュメントはタスクの複雑度に応じて分けられます。極小タスクでは既存ルールに従えば十分です。小〜中規模タスクでは関連モジュール文書を読む必要があります。中〜複雑なタスクでは、先に調査して短い方針を書くべきです。モジュール横断のリファクタリングでは、完全なspec、タスク分解、チェックリスト、アーカイブが必要になります。

知識管理用のAgentは、目次の維持、古い文書の検出、調査結果の再利用可能な資料化、更新すべき知識の通知を担当できます。これにより文書はタスク終了時に受け身で追記されるものではなく、プロセスの一部になります。

## 4. Skill層

Skillは「どう作業するか」を固定する仕組みです。単なるプロンプトではなく、再利用できるプロセステンプレートです。

ドメインSkillは、AIにプロジェクトの中核概念、よく使うディレクトリ、破ってはいけない依存関係、避けるべき過去の落とし穴を伝えます。Review系Skillは、コードレビュー、文書レビュー、コンテキスト改善を標準化します。Super Mode系Skillは、要件整理、調査、実装、確認、アーカイブを一連の流れとしてつなぎます。Trellis Skillは、長期タスクを追跡可能で再開可能なタスク列に分解します。

学習補助Skillも価値があります。ひとつのタスクで得た新しい発見を「次回すぐ使えるルール」に変換できるため、AI協働は現在の要件を満たすだけでなく、少しずつ賢くなっていきます。

## 5. コマンド層

コマンド層は、よく使うプロセスを明確な入口にします。たとえば、完全な要件整理フローを開始するコマンド、未完了のTrellisタスクを再開するコマンド、終了前に確認、アーカイブ、要約を実行するコマンドなどです。

コマンドの意味は、数行の入力を減らすことではありません。手順漏れを減らすことです。複雑なタスクでは、会話の途中で状態が失われがちです。調査したのにspecを更新しない、コードを書いたのにテストしない、実装を変えたのに経験を記録しない。コマンド入口は、こうした手順を固定してくれます。

## 6. Trellisワークフロー層

Trellisの核心的な価値は、タスクを一回限りの会話から切り離し、永続化、再開、レビューが可能な作業単位にすることです。

通常、三つのモデルを持ちます。

- Workflow layer：計画、実行、完了までの段階を定義します。
- Persistence layer：タスク、spec、コンテキスト、確認結果をファイルに保存します。
- Platform integration layer：コマンド、hooks、Agentを通じて実際の開発環境に流れを組み込みます。

Plan段階では、AIは目標を明確にし、コンテキストを確認し、タスクを分解し、成功基準を書きます。Execute段階では、タスクに沿って段階的に実装し、中間状態をworkspaceに記録します。Finish段階では、テストを実行し、reviewを行い、specを更新し、経験をアーカイブして、コミットの準備をします。

`.trellis/tasks/`はタスク状態の記録に、`.trellis/spec/`は要件、設計、意思決定の記録に、`.trellis/workspace/`は実行中の中間状態、調査結果、review dossierの保存に向いています。これにより長期タスクはチャットウィンドウのコンテキスト長に縛られません。

## 7. Subagent分担層

Subagentは、並行で独立しており、コンテキスト量が多いタスクに向いています。

実装Agentは明確なタスクを受け取って局所的な変更を行えます。チェックAgentは別の視点から変更をレビューできます。調査Agentは大量のファイルを読んで要約を返し、主会話のコンテキストを汚さずに済みます。

この分担の最大の価値は、注意の隔離です。主会話は意思決定と統合を担当し、子Agentは局所調査、実装、レビューを担当します。モジュール横断タスクでは、一つの会話がすべてを抱え込むよりも安定します。

ただし、Subagentにも制約が必要です。タスク境界は明確であるべきです。入力と出力も明確にすべきです。複数のAgentが同じ共有状態を同時に変更してはいけません。そうしないと、並行処理が衝突の原因になります。

## 8. 自動注入とHook層

Hooksは「毎回覚えておくべきこと」を自動動作に変えられます。

たとえば、session-start hookは会話開始時に現在のタスク状態、重要文書の索引、最近の作業記録を注入できます。ワークフロー状態注入スクリプトは、AIが計画、実行、完了のどの段階にいるかを思い出させます。Subagentコンテキスト注入スクリプトは、子Agentにタスク境界、読める資料、出力形式を伝えます。

エディタhooksも組み合わせられます。プロジェクトを開くとき、ファイル保存時、コミット前などに確認情報を注入し、AIプログラミングとローカル開発フローをひとつの流れに近づけられます。

自動注入の原則は、安定し、必要で、短い情報だけを注入することです。すべての文書をコンテキストに詰め込むべきではありません。多すぎるコンテキストは焦点を薄め、誤読のリスクも上げます。

## 9. 品質ゲート層

コードを書き終えたことは、タスク完了を意味しません。成熟したAIプログラミング体系には、明確な品質ゲートが必要です。

一般的なゲートには、関連テストを実行したか、重要パスを検証したか、lintやフォーマットの問題がないか、文書を更新したか、再利用を確認したか、既知リスクを記録したか、機密情報を漏らしていないか、などがあります。

Review dossierは優れた完了ツールです。今回の変更範囲、テスト証拠、リスク、未完了事項、レビュー結論を記録できます。後から人間がレビューする場合でも、AIが引き継ぐ場合でも、現在の状態を素早く理解できます。

## 10. Gitとコミット戦略

コミット前チェックは「追跡可能」と「巻き戻せる」を中心に考えるべきです。コミットメッセージは、何を変えたかだけでなく、なぜ変えたかを説明します。ステージングでは具体的なファイルを選び、ローカル設定、キャッシュ、一時ファイル、機密情報を一緒に含めないようにします。

AI協働において、コミット戦略にはさらに別の価値があります。それは、一回の会話の成果をレビュー可能な境界として固定することです。よいコミットは、後続の人間やAIが変更意図、検証方法、影響範囲を理解できるものです。

## 11. この体系の強み

第一に、コンテキストの出所が明確です。AIはプロジェクトルールを空想で補う必要がなく、入口ファイル、知識文書、Skill、タスク状態から段階的にコンテキストを復元できます。

第二に、プロセスが完整です。要件整理から実装、確認、アーカイブまで対応する仕組みがあり、「書き終えたら完了」という状況を減らせます。

第三に、複雑なプロジェクトに向いています。複数モジュール、複数ホスト、複数タスクが並行する場合でも、Trellis、Subagent、review dossierが複雑さを分解します。

第四に、経験を蓄積できます。調査と振り返りは一度のチャットに残るだけでなく、プロジェクト文書とspecに蓄積されます。

## 12. 注意すべき点

第一に、設定ファイルを空の殻にしないことです。settingsやhooksが実際の流れに接続されていなければ、自動化は名目上のものにすぎません。

第二に、文書ディレクトリには索引が必要です。索引がないと、AIはどの文書を先に読むべきか判断しにくく、重要な資料を見落としやすくなります。

第三に、specを過度にテンプレート化しないことです。テンプレートは構造を保証しますが、本当に価値があるのは具体的な制約、実際の意思決定、明確な受け入れ基準です。

第四に、複数のフローが重複する可能性があります。Super Mode、Trellis、Skill、コマンド、hooksはいずれもプロセスを管理できます。境界が曖昧だと、重複したリマインドや上書きが起こります。よりよい方法は、誰が計画を担当し、誰が実行し、誰が確認し、誰がアーカイブするかを明確にすることです。

## 13. 推奨される使い方

新機能開発では、まずAIに入口ルールと関連モジュール文書を読ませ、短いspecを書き、タスクを分解し、タスクごとに実装と検証を進めます。

Bug修正では、先に問題を再現し、関連コードと履歴を調べ、最小変更で修正し、回帰テストまたは検証記録を追加します。

コードレビューでは、まずreview dossierを作り、レビューAgentまたはレビューSkillに、正しさ、境界、再利用、テスト、リスクを順番に確認させます。

文書レビューでは、文章が自然かだけでなく、次回タスクのコンテキスト復元に役立つかを確認します。背景、制約、入口ファイル、検証方法、未決事項があるかを見るべきです。

小さな素早い変更では、完全なフローを起動する必要はありません。それでも、境界を確認する、最小変更を行う、結果を検証する、という三つの動作は保つべきです。

## 14. まとめ

よいAIプログラミング体系は、本質的に三つの問いに答えます。AIはどこからプロジェクトを理解するのか。AIはどの手順で作業するのか。AIの経験はどのように蓄積されるのか。

入口ドキュメントは「どこから理解するか」に答えます。Skill、コマンド、Trellisは「どの手順で作業するか」に答えます。文書、spec、review dossier、アーカイブ機構は「どのように蓄積するか」に答えます。これらの層が組み合わさると、AIは一回の会話のコード生成器ではなく、継続的に協働でき、検査可能で、再開可能なエンジニアリングパートナーに近づきます。
