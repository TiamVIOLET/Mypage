## 1. 背景与整体定位

在一个复杂业务项目里，AI 编程不能只依赖一次性的提示词。更稳定的做法，是把项目规则、领域知识、技能系统、任务工作流、Subagent 分工、自动上下文注入、质量门禁与知识沉淀组合成一套协作体系。

这套体系的目标很明确：让 AI 先理解项目再写代码；让 AI 按流程推进而不是即兴发挥；让复杂任务可以拆分、恢复和追踪；让调研、复盘和约定持续沉淀为项目资产。

整体上，它可以分成几层：

```text
AI 编程体系
├── 项目入口层：CLAUDE.md / AGENTS.md
├── 知识文档层：.claude/docs / 模块 CLAUDE.md
├── 技能层：.claude/skills
├── 命令层：.claude/commands
├── Agent 分工层：.claude/agents
├── Trellis 工作流层：workflow / tasks / spec / workspace
├── 自动注入层：hooks / settings / 编辑器 hooks
└── 质量与归档层：review / test / update-spec / commit / finish-work
```

## 2. 项目入口层

项目入口层负责告诉 AI：这个项目是什么、边界在哪里、应该遵循哪些全局规则。

根目录 `CLAUDE.md` 通常承担三类职责。第一是项目架构说明，例如技术栈、运行方式、核心模块和关键业务对象。第二是分层边界说明，例如哪些目录是应用宿主，哪些目录是共享层，哪些依赖关系被允许，哪些依赖关系必须禁止。第三是 AI 工作流约束，例如不要直接猜测需求、优先复用既有模式、改动前先查文档、完成后要更新相关知识。

模块级 `CLAUDE.md` 则更适合描述局部规则。比如某个模块的命名方式、目录结构、常用服务、测试入口和历史约定。这样 AI 进入某个子目录时，可以获得比全局规则更精确的上下文。

`AGENTS.md` 可以作为跨工具的补充入口。不同编码代理读取的默认文件不完全相同，因此把关键规则同步到通用入口，可以减少模型切换或工具切换时的上下文丢失。

## 3. 知识文档层

`.claude/docs/` 用来存放稳定的项目知识：架构说明、模块地图、常见问题、调研记录、复盘结果和历史决策。它的价值不只是“给人看”，更重要的是让 AI 在下一次任务中能快速恢复上下文。

文档可以按任务复杂度分级：极小任务只需要遵循现有规则；小到中等任务需要读相关模块文档；中等到复杂任务需要先做调研并写简短方案；跨模块重构则需要完整 spec、任务拆分、检查清单和归档。

知识管理员类 Agent 可以负责维护目录索引、发现过期文档、把调研结果沉淀成可复用资料，并提醒开发者哪些知识应该更新。这样文档不会只在任务结束时被动补写，而是成为流程的一部分。

## 4. Skill 技能层

Skill 是把“怎么做事”固定下来的机制。它不只是提示词，而是可复用的过程模板。

业务领域 Skill 可以告诉 AI：当前项目的核心概念是什么，常见目录在哪里，哪些依赖关系不能违反，哪些历史坑需要避开。Review 类 Skill 可以把代码审查、文档审查和上下文改进流程标准化。Super Mode 类 Skill 可以把需求澄清、调研、实现、检查和归档串起来。Trellis Skill 则负责把长期任务拆成可追踪、可恢复的任务链。

学习辅助 Skill 也很有价值。它可以把一次任务里的新发现提炼成“下次可以直接使用的规则”，让 AI 协作不只是完成当前需求，而是在不断变聪明。

## 5. 命令层

命令层负责把常用流程变成明确入口。比如一个命令可以启动完整的需求澄清流程，一个命令可以继续未完成的 Trellis 任务，一个命令可以在结束前执行检查、归档和总结。

命令的意义不是减少几行文字输入，而是减少流程遗漏。复杂任务很容易在对话中丢掉状态：做了调研但没更新 spec，写了代码但没跑测试，改了实现但忘了记录经验。命令入口可以把这些步骤固定下来。

## 6. Trellis 工作流层

Trellis 的核心价值是把任务从一次性对话中抽离出来，变成可持久化、可恢复、可审查的工作单元。

它通常包含三层模型：

- Workflow layer：定义从计划、执行到收尾的阶段。
- Persistence layer：把任务、spec、上下文、检查结果写入文件。
- Platform integration layer：通过命令、hooks 和 Agent 把流程嵌入实际开发环境。

在 Plan 阶段，AI 需要澄清目标、检查上下文、拆分任务并写下成功标准。在 Execute 阶段，AI 根据任务逐步实现，并把中间状态写入 workspace。在 Finish 阶段，AI 要运行测试、做 review、更新 spec、归档经验并准备提交。

`.trellis/tasks/` 适合记录任务状态，`.trellis/spec/` 适合记录需求、设计和决策，`.trellis/workspace/` 适合保存执行中间态、调研结果和 review dossier。这样长期任务不会被聊天窗口的上下文长度绑架。

## 7. Subagent 分工层

Subagent 适合处理并行、独立、上下文较重的任务。

实现类 Agent 可以接收明确任务并做局部修改；检查类 Agent 可以从另一个角度审查改动；调研类 Agent 可以阅读大量文件后输出摘要，而不污染主会话上下文。

这种分工最大的价值是隔离注意力。主会话负责决策和整合，子代理负责局部调查、实现或审查。对于跨模块任务，这比让一个对话同时承担所有工作更稳。

不过 Subagent 也需要约束：任务边界要清楚，输入输出要明确，不能让多个 Agent 同时修改同一片共享状态。否则并行会变成冲突源。

## 8. 自动注入与 Hook 层

Hooks 可以把“应该每次都记得的事”变成自动行为。

例如，session-start hook 可以在会话开始时注入当前任务状态、关键文档索引和最近的工作记录。工作流状态注入脚本可以提醒 AI 当前处于计划、执行还是收尾阶段。Subagent 上下文注入脚本可以让子代理知道自己的任务边界、可读资料和输出格式。

编辑器 hooks 也可以配合使用。比如在打开项目、保存文件或提交前注入检查信息，让 AI 编程和本地开发流程更接近一个整体。

自动注入的原则是：注入稳定、必要、短小的信息，不要把所有文档都塞进上下文。太多上下文会稀释重点，也会增加误读风险。

## 9. 质量门禁层

代码写完不等于任务完成。一个成熟的 AI 编程体系需要明确的质量门禁。

常见门禁包括：相关测试是否运行、关键路径是否验证、是否有 lint 或格式问题、是否更新文档、是否检查复用、是否记录已知风险、是否确认没有泄露敏感信息。

Review dossier 是一个很好的收尾工具。它可以记录本次改动范围、测试证据、风险点、未完成事项和审查结论。后续无论是人类审查，还是 AI 继续接手，都能快速知道当前状态。

## 10. Git 与提交策略

提交前检查应该围绕“可追溯”和“可回滚”。提交信息要说明为什么改，而不只是列出改了什么。暂存文件时应尽量选择具体文件，避免把本地配置、缓存、临时文件或敏感信息一起提交。

对于 AI 协作来说，提交策略还有一个额外价值：它把一次对话的结果固定成可审查的边界。好的提交应当能让后续的人和 AI 都看懂改动意图、验证方式和影响范围。

## 11. 当前体系的优势

第一，上下文来源清晰。AI 不需要凭空猜测项目规则，而是能从入口文件、知识文档、Skill 和任务状态中逐步恢复上下文。

第二，流程完整。从需求澄清到实现、检查、归档都有对应机制，减少“写完就算结束”的情况。

第三，适合复杂项目。多模块、多宿主、多任务并行时，Trellis、Subagent 和 review dossier 可以把复杂度拆开。

第四，支持经验积累。调研和复盘不再只留在一次聊天里，而是沉淀到项目文档和 spec 中。

## 12. 需要注意的问题

第一，配置文件不能只是空壳。如果 settings 或 hooks 没有真正接入流程，自动化就只是名义上的自动化。

第二，文档目录需要索引。没有目录索引时，AI 很难判断应该先读哪份文档，也容易遗漏关键资料。

第三，spec 不能过度模板化。模板可以保证结构，但真正有价值的是具体约束、真实决策和明确验收标准。

第四，多套流程可能重叠。Super Mode、Trellis、Skill、命令和 hooks 都能管理流程，如果边界不清，就会产生重复提醒和互相覆盖。更好的做法是明确：谁负责计划，谁负责执行，谁负责检查，谁负责归档。

## 13. 推荐使用方式

新功能开发时，先让 AI 读取入口规则和相关模块文档，再写简短 spec，拆分任务，最后按任务逐步实现和验证。

Bug 修复时，先复现问题，再查相关代码和历史记录，然后用最小改动修复，并补充回归测试或验证记录。

代码审查时，先生成 review dossier，再让审查 Agent 或审查 Skill 按正确性、边界、复用、测试和风险逐项检查。

文档审查时，不只看语句是否通顺，还要看是否能帮助下一次任务恢复上下文：有没有背景、约束、入口文件、验证方式和未决问题。

快速小改时，也不需要启动完整流程，但仍应保持三个动作：确认边界、做最小修改、验证结果。

## 14. 总结

一套好的 AI 编程体系，本质上是在回答三个问题：AI 从哪里理解项目？AI 按什么流程工作？AI 的经验如何沉淀下来？

入口文档解决“从哪里理解”，Skill、命令和 Trellis 解决“按什么流程工作”，文档、spec、review dossier 和归档机制解决“如何沉淀”。当这些层组合起来，AI 就不再只是一次对话里的代码生成器，而会更像一个可以持续协作、可检查、可恢复的工程伙伴。
