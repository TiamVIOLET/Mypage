## プロジェクト概要

AI-Girlfriend は、バーチャルコンパニオン、音声対話、AI VTuber といったシーンに向けたキャラクター実行システムです。単に大規模言語モデルをチャット欄につなぐものではなく、**リアルタイム音声インタラクション、Live2D キャラクター表現、長期記憶、動的人格、Agent 実行能力、ツール呼び出し**をひとつの完全な流れとして統合し、AI キャラクターにより高い継続性、表現力、拡張性を持たせています。

コード構成を見ると、バックエンドは FastAPI と WebSocket をベースに構築され、フロントエンドはインタラクション画面と Live2D 表示を担当します。一方、バックエンドは ASR、LLM / Agent、TTS、VAD、MCP ツール、会話履歴、キャラクター設定などのコア機能を統括します。プロジェクト全体はモジュール化された設計を採用しており、各機能はファクトリーパターンと設定システムを通じて接続されるため、二次開発にも適しています。

---

## コアコンセプト：性格があり、記憶を持ち、行動できる AI キャラクターシステム

AI-Girlfriend の中核的な価値は、単に「チャットできる」ことではありません。AI キャラクターを、より継続的に存在するバーチャルキャラクターらしくすることにあります。

- **性格がある**：Personality Engine によってキャラクター状態を維持し、口調、表現方法、インタラクション方針を会話に応じて動的に変化させます。
- **記憶がある**：Character Memory によって長期情報を保存・検索し、単なるチャット履歴の再生に留まりません。
- **行動できる**：Agent システムと MCP ツール呼び出しにより、キャラクターが外部機能に接続でき、テキスト出力だけに限定されません。
- **表現できる**：音声、Live2D 表情、リアルタイムインタラクションを通じて、コンパニオンアプリに近い体験を実現します。

したがって、これは通常のチャットボット Demo というより、「AI キャラクターランタイム」として理解するのがより適切です。

## 機能モジュール説明

### 1. リアルタイムインタラクションと WebSocket 通信

このプロジェクトは FastAPI でサーバーサイドを構築し、WebSocket を通じてフロントエンドとリアルタイム通信を維持します。WebSocket 層は、ユーザー入力、音声データ、制御信号、設定切り替えリクエストを受け取り、同時にテキスト、音声、Live2D モデル情報、セッション状態などをフロントエンドへプッシュします。

主な機能は次のとおりです。

- クライアント接続の確立と維持。
- テキスト入力とマイク音声データの受信。
- 会話トリガー、自発発話、割り込み信号のサポート。
- 設定切り替え、履歴読み取り、キャラクター状態同期のサポート。
- 1対1セッションとグループセッションのサポート。
- 静的リソースサービスを通じたフロントエンド、Live2D モデル、アバター、背景、キャッシュ音声の提供。

この層はプロジェクト全体の通信ハブに相当し、フロントエンドのインタラクションとバックエンドのインテリジェント機能をつなぎます。

---

### 2. 音声入力 ASR モジュール

ASR モジュールはユーザー音声をテキストに変換する役割を持ち、音声インタラクション経路の入口です。このプロジェクトは複数の音声認識バックエンドをサポートしており、ローカルモデルもクラウドサービスも利用できます。

サポート済み、または準備されている ASR 方式は次のとおりです。

- sherpa-onnx；
- FunASR；
- Faster-Whisper；
- Whisper.cpp；
- OpenAI Whisper；
- Groq Whisper；
- Azure ASR。

ASR は設定ファイルによって具体的な実装を選択し、バックエンドはファクトリークラスを通じて対応するエンジンを生成します。この設計により、認識モデルの差し替えコストを下げ、デバイス性能、言語シーン、遅延要件に応じた選択がしやすくなっています。

---

### 3. 音声出力 TTS モジュール

TTS モジュールは AI の返信を音声に変換する役割を持ち、キャラクターが「話す」ための重要な部分です。このプロジェクトは豊富な音声合成バックエンドをサポートしており、軽量な方式から、より高品質またはキャラクター性の強い声色を持つ TTS システムまで接続できます。

サポート済み、または準備されている TTS 方式は次のとおりです。

- sherpa-onnx TTS；
- Edge TTS；
- pyttsx3；
- MeloTTS；
- Coqui-TTS；
- GPT-SoVITS；
- CosyVoice / CosyVoice2；
- XTTS；
- Bark；
- Fish Audio；
- MiniMax TTS；
- Azure TTS；
- OpenAI TTS；
- SparkTTS；
- SiliconFlow TTS。

TTS は単なる最後の「テキスト読み上げ」ではありません。ストリーミング出力、文分割、最初の一文の高速応答、音声キャッシュと連携し、ユーザーが感じる応答速度やキャラクターのリアリティに影響します。

---

### 4. VAD 音声活動検出

VAD は、ユーザーがいつ話しているか、いつ話し終えたかを検出するために使われます。ボタンを押さない音声インタラクションや、割り込み体験の重要な基盤です。

プロジェクトでは主に Silero VAD を使用しています。その役割は次のとおりです。

- マイク入力に有効な音声が存在するかを判定する。
- ユーザー音声の終了タイミングを補助的に認識する。
- より自然な連続会話をサポートする。
- 割り込み機構と連携し、ユーザーと AI が同時に話すことで生じる体験上の問題を減らす。

バーチャルコンパニオン系アプリにとって、VAD の意味は非常に大きいです。毎回手動でボタンをクリックするのではなく、より人間同士の会話に近い方法でキャラクターとやり取りできるようになります。

---

### 5. LLM と Agent システム

LLM と Agent システムはプロジェクトの知能の中核です。このプロジェクトは単一のモデル提供元に固定されておらず、アダプター層を通じて複数の LLM バックエンドに接続し、その上で異なる種類の Agent をラップしています。

サポート、または互換性のある LLM の方向性は次のとおりです。

- Ollama；
- Claude；
- OpenAI / OpenAI-compatible API；
- Gemini；
- Zhipu AI；
- DeepSeek；
- Groq；
- Mistral；
- LM Studio；
- llama.cpp。

Agent 層は、モデル呼び出し、コンテキスト管理、キャラクタープロンプト、ツール呼び出し、出力処理を整理する役割を担います。プロジェクトには BasicMemoryAgent、GenericAgent、HybridAgent、HumeAIAgent、LettaAgent などの異なる実装が含まれています。

その中でも HybridAgent は重要です。雑談型のキャラクター能力とタスク実行能力を分離し、ひとつの Agent が日常的な陪伴やキャラクター表現を担当し、もうひとつの Agent がより実行寄りのタスクを担当できます。これにより、プロジェクトは単に「チャットできる」だけでなく、デスクトップアシスタント、タスクエージェント、ツール型キャラクターへ拡張する基盤も備えています。

---

### 6. Personality Engine 動的人格システム

Personality Engine は、このプロジェクトを通常のキャラクター Prompt と区別するコアモジュールのひとつです。一般的な AI キャラクターは固定された人物設定プロンプトに依存しがちですが、このプロジェクトでは人格を実行中の状態システムとして扱っています。

これは、たとえば次のような複数のキャラクター状態量を維持します。

- vitality；
- stress load；
- trust score；
- resentment score；
- session state；
- turn count。

同時に、ego、persona などの人格ベクトルによってキャラクターの内的傾向と外的表現を記述し、会話内容に応じてそのターンの personality overlay を生成します。これにより、キャラクターの次の要素に影響を与えます。

- 口調 tone；
- 語彙スタイル lexical style；
- 文型 sentence form；
- コントロール感 control style；
- 感情温度 emotional temperature；
- 開始と締め方；
- マスク強度と本来の人格が漏れ出す度合い。

これにより、キャラクターは機械的に人物設定を適用するだけではなく、インタラクションに応じて徐々に異なる状態を見せられるようになります。長期的な陪伴、ロールプレイ、バーチャル人格アプリケーションにとって、このモジュールは非常に価値のある部分です。

---

### 7. Character Memory 長期記憶システム

Character Memory は、このプロジェクトのもうひとつのコア機能です。単純にチャット履歴を保存するのではなく、キャラクターの記憶を構造化され、検索可能で、書き戻し可能な長期知識レイヤーとして扱います。

このモジュールは SQLite を永続化ストレージとして使用し、FTS 検索をサポートします。FTS が利用できない場合は、通常検索へフォールバックすることもできます。記憶内容は次のように整理されます。

- entities：エンティティ。たとえばユーザー、キャラクター、場所、重要なオブジェクト。
- triples：三つ組関係。たとえば「ユーザー - 好き - ある物事」。
- attributes：エンティティ属性。たとえば好み、身分、状態。
- episodic events：エピソード型記憶。重要なインタラクションの断片を保持するために使います。

会話開始前、システムはユーザー入力に基づいて関連する記憶を検索し、memory packet として構成して現在のターンに注入します。返信終了後には、新しい情報を記憶庫に書き戻すこともできます。これにより、キャラクターはユーザー、関係性、共有した経験に関する情報を徐々に蓄積できます。

これは、通常のチャットボットにおける最も明白な問題、つまり毎回の会話が初対面のようになってしまう問題を解決します。長期記憶によって、キャラクターはより自然に関係性の感覚を継続できます。

---

### 8. MCP ツール呼び出しシステム

MCP モジュールは、プロジェクトにツール呼び出し能力を提供し、AI キャラクターが外部ツールやサービスに接続できるようにします。プロジェクトには Server Registry、Tool Adapter、Tool Manager、MCP Client、Tool Executor、JSON Detector などのコンポーネントが含まれています。

大まかな流れは次のとおりです。

1. 設定に基づいて有効化された MCP server を読み込む。
2. Tool Adapter を通じてツール情報を取得し、モデルが使用しやすいツールプロンプトを生成する。
3. Tool Manager がツール定義を管理する。
4. MCP Client が外部 MCP server と通信する。
5. Tool Executor がツール呼び出しを実行する。
6. Agent がツール結果を整理し、キャラクターの口調に合った自然な返信に変換する。

この仕組みにより、キャラクターはテキストを生成するだけでなく、検索、実行、サービス呼び出しを行い、その結果をキャラクターらしい表現へ変換できます。AI キャラクターを「寄り添うだけでなく、作業もできる」アプリケーションにしたい場合、MCP は重要な拡張ポイントです。

---

### 9. Live2D キャラクター表現層

Live2D モジュールはキャラクターの視覚表現を担当します。プロジェクトは `live2d-models/` 配下のモデルリソースを読み込み、キャラクター設定とモデル、表情、モーションなどを紐づけることができます。

関連機能は次のとおりです。

- Live2D モデル設定の管理。
- キャラクターごとのモデル切り替え。
- キーワードマッピングによる表情トリガー。
- 音声出力と連携した、より完全なキャラクター表現。
- アバター、背景、フロントエンド表示リソースのサポート。

このプロジェクトにおける Live2D は単なる装飾ではなく、キャラクター表現層の一部です。音声、人格、記憶と組み合わさることで、ユーザーはより強い「キャラクターがそこにいる感覚」を得られます。

---

### 10. 複数キャラクターと設定管理

プロジェクトは設定システムを通じて、異なるキャラクターや異なる機能バックエンドを管理します。メイン設定ファイルは `conf.yaml` で、デフォルトテンプレートは `config_templates/` に、キャラクターごとの上書き設定は `characters/` ディレクトリに配置されます。

設定内容は非常に広範囲にわたります。

- システムのリッスンアドレスとポート。
- キャラクター名、アバター、Live2D モデル。
- persona prompt。
- ASR / TTS / VAD 設定。
- LLM と Agent 設定。
- Character Memory 設定。
- Personality Engine 設定。
- MCP ツール設定。
- 翻訳、配信、その他の拡張機能。

この設定ベースの設計により、キャラクターを素早く切り替えたり、キャラクターごとに異なる声、モデル、記憶庫、人格パラメータを割り当てたりできます。

---

### 11. チャット履歴とセッション管理

プロジェクトにはチャット履歴管理機能が組み込まれており、履歴セッションの作成、読み取り、切り替え、削除ができます。各 WebSocket 接続は独立した ServiceContext を持ち、現在のセッションに必要な設定、モデル、Agent、ASR、TTS、VAD、記憶、人格状態を管理します。

セッションシステムは次をサポートします。

- 1対1の会話。
- グループ会話。
- 自発発話。
- ユーザーによる割り込み。
- 画像入力。
- 履歴メッセージの永続化。
- 中断時に、すでに聞こえた AI 返信を保存すること。
- 人格状態の履歴メタデータへの書き込み。

この部分により、プロジェクトは一回限りの問答ではなく、複数ターン、複数セッション、複数キャラクターのインタラクションを維持できる実行システムになっています。

---

### 12. 翻訳、配信、拡張機能

コアの対話経路に加えて、プロジェクトにはいくつかの拡張機能も含まれています。

- DeepLX / Tencent 翻訳。
- Bilibili Live 接続。
- Proxy WebSocket モード。
- 画像入力サポート。
- Web Tool。
- ローカルモデルとクラウド API のハイブリッド接続。

これらの機能により、プロジェクトはより多くのシーンへ拡張できます。たとえば、配信インタラクション用キャラクターは弾幕に接続できます。多言語キャラクターは翻訳モジュールを通じて言語をまたぐ入出力を処理できます。プロキシモードを使えば、外部クライアントが同じバックエンド機能に接続できます。

## 実行フロー

1回の完全なインタラクションとして見ると、AI-Girlfriend のコア実行フローは次のように要約できます。

```text
ユーザー入力
  → テキスト入力 / マイク音声
  → VAD が音声活動を判定
  → ASR が音声をテキストに変換
  → Character Memory が関連する長期記憶を検索
  → Personality Engine が動的人格オーバーレイを生成
  → Agent / LLM が返信を生成
  → MCP ツール呼び出し（任意）
  → TTS が返信を音声に変換
  → Live2D 表情とフロントエンド表示が連動
  → チャット履歴と長期記憶へ書き戻し
```

このフローはプロジェクトの完全性を示しています。単一機能の寄せ集めではなく、入力、理解、記憶、人格、生成、実行、音声、表現層をひとつのキャラクターシステムとして接続しています。

## 適用シーン

AI-Girlfriend は次のような方向性に適しています。

- AI 彼女 / 彼氏 / バーチャルコンパニオンアプリ。
- 長期記憶を持つロールプレイシステム。
- 音声対話可能な Live2D デスクトップパートナー。
- AI VTuber または配信インタラクション用キャラクター。
- ローカル優先のプライベート音声 AI アプリ。
- 人物設定を持つデスクトップアシスタント。
- ツール呼び出し能力を備えたキャラクター型 Agent。
- 複数キャラクター、複数設定に対応するバーチャル人格実験プラットフォーム。

目的が単に問答ロボットを作ることだけであれば、このプロジェクトは完成度が高すぎる、あるいはやや重く感じられるかもしれません。しかし目的が「継続的に存在する AI キャラクター」を作ることであれば、そのアーキテクチャはより大きな強みを発揮します。

## プロジェクトの強み

### 1. 高いモジュール性

ASR、TTS、LLM、VAD、Agent、MCP、翻訳などの機能は、比較的独立したモジュールとファクトリークラスを持っており、バックエンドの差し替えや新しい実装の追加が容易です。

### 2. キャラクター感が強い

Personality Engine、Character Memory、Live2D、音声出力が組み合わさることで、キャラクターの継続性と表現力が強化され、通常のテキストチャットボットよりも陪伴系シーンに適しています。

### 3. ローカル実行も、ハイブリッドデプロイも可能

プロジェクトはローカルモデルとクラウド API のハイブリッド接続をサポートしています。開発者はコスト、プライバシー、性能、モデル品質に応じて、柔軟にデプロイ方式を選択できます。

### 4. 拡張方向が明確

新しいモデルの追加、新しい TTS の接続、MCP ツールの拡張、キャラクター設定の追加、人格戦略の調整を行う場合でも、既存のディレクトリ構造に沿って開発を続けられ、作り直す必要がありません。

### 5. 完全なランタイム思想をすでに備えている

入力から出力まで、キャラクター状態から記憶の書き戻しまで、フロントエンド表現からバックエンドのツール呼び出しまで、完全な流れをカバーしており、二次開発の土台として適しています。

## クイックスタート

プロジェクトは uv で依存関係を管理します。よく使うコマンドは次のとおりです。

```bash
uv sync
```

```bash
uv run run_server.py
```

```bash
uv run run_server.py --verbose
```

```bash
uv run upgrade.py
```

```bash
ruff check .
```

## まとめ

AI-Girlfriend の重点は「多くのモデルにつながっている」ことではなく、AI キャラクターに必要な主要機能をすでにひとつの完全なシステムとして整理している点にあります。

このプロジェクトにより、キャラクターは単に次のことができるだけではありません。

- 質問に答えられる。
- 音声を再生できる。
- Live2D の姿を表示できる。

さらに次の能力を備えます。

- 動的に変化する人格状態。
- 検索可能で書き戻し可能な長期記憶。
- 拡張可能な Agent とツール呼び出し能力。
- リアルタイム陪伴体験に向けた音声・視覚表現フロー。

したがって、このプロジェクトは AI Companion、AI VTuber、バーチャルキャラクターアシスタント、長期インタラクション型キャラクターシステムの開発基盤としてより適しています。AI を「チャットモデル」から「継続的なキャラクター」へ進めたい開発者にとって、AI-Girlfriend は深く研究し、二次開発する価値のあるプロジェクトです。
